1. Context and Problem Statement
ADR-007 defines the Person node schema and establishes a federated profile construction approach where multiple authority sources (DPRR, Wikidata, VIAF, FAST, Nomisma, Trismegistos, LGPN) each contribute claims to a single person profile. The question this ADR addresses is: how much of the harvest pipeline should be agent-driven reasoning versus deterministic rule execution?
The naive answer — "use agents for everything because federations are complex" — introduces unnecessary non-determinism into operations that are actually mechanical. The opposite extreme — "use deterministic rules throughout" — breaks immediately on the core problem federations create: the same person expressed in radically different name forms across sources that have no shared identifier.
The name reconciliation problem
Consider Pompey (DPRR 1976) across the seven operational federation sources:

No deterministic string matcher, edit-distance scorer, or lookup table reliably unifies these. The cognomen is the primary identifier in some sources; the praenomen in others; the Latin form in some, transliterated Greek in one, a bare slug in another. An agent reasoning over the full name surface, the date window, and the DPRR ID cross-link is the only reliable reconciler across this surface area.
However: once the agent concludes these records all refer to POMP1976, the merge is arithmetic. The agent decides what; a deterministic writer executes how. This distinction drives the entire architecture.
2. Decision
The Person Harvest Agent operates on a three-layer architecture: deterministic pre-processing feeds structured input to agent reasoning, which hands structured decisions to deterministic execution. The agent operates only in the middle layer and never writes directly to the graph.

3. Layer 1 — Deterministic Pre-Processing
Layer 1 transforms raw source data into normalised, structured inputs. No judgment calls occur here. Each operation has a defined rule and a defined output schema.

3.4  DPRR ID exact match
If two source records both carry the same dprr_id, they resolve to the same person node. This is a hash lookup — no agent involvement. The DPRR ID is the strongest identity anchor in the corpus and takes precedence over all name-based matching.
4. Layer 2 — Agent Reasoning
The agent receives the structured output of Layer 1 and produces a PersonHarvestPlan: a complete, serialised record of every decision made during the reasoning phase. No graph writes occur during Layer 2. The plan is auditable, resumable, and version-controlled.
4.1  Tasks requiring agent reasoning

Agent approach
Cross-federation name reconciliation

Distinguishing precision gap from soft conflict from hard conflict requires understanding of what each source is expected to cover

Authority tier weighting
Source authority varies by attribute — DPRR outranks Wikidata on offices but Wikidata may be more current on cross-IDs

DPRR filiation strings have edge cases: unknown praenomina ("-. f."), identical names across generations, maternal vs paternal ambiguity

Wikidata has no single P31 value for "legendary historical figure"; Romulus has P31=human in some claims, legendary character in others

4.3  Agent constraints
The agent operating in Layer 2 is explicitly prohibited from:
Writing any node or relationship directly to the graph
Evaluating numeric confidence thresholds (D10 decision table is Layer 3)
Generating freeform Cypher
Making promotion decisions (ApprovalRequired policy; human gate)
Querying the live graph during reasoning — all graph state needed for reasoning is pre-fetched and provided as structured context before the agent is invoked
The last constraint is important: the agent receives a context packet, not live graph access. This makes the reasoning deterministic relative to a fixed graph snapshot, prevents the agent from making writes through tool calls, and enables the plan to be replayed or audited against the same inputs.

7. Family Tree Construction Strategy
ADR-007 establishes a person-by-person construction approach: each person is fully enriched (including full ancestry) before moving to the next. This section defines the traversal rules and stopping conditions.
7.1  Traversal order
Construction proceeds depth-first along paternal lines (FATHER_OF chains) because DPRR filiation is predominantly paternal and the gens network is the primary analytical axis. Maternal lines (MOTHER_OF) are pursued to the same depth but may terminate earlier where DPRR records are thinner.
Traversal priority
Rationale
1. Paternal ancestor chain (FATHER_OF recursive)
DPRR filiation is primarily paternal; gens identity follows paternal line
2. Maternal ancestor chain (MOTHER_OF recursive)
Maternal lines important for adoption, alliance, and inheritance analysis
3. Spouses (SPOUSE_OF)
Each spouse may introduce a new gens subgraph worth traversing
4. Siblings (SIBLING_OF)
Enriched after parents to avoid duplicate parent resolution
5. Children (FATHER_OF / MOTHER_OF outbound)
Enriched last; descendants are lower priority than ancestors for prosopography

8. Consequences
8.1  Positive
Agent reasoning is front-loaded and fully auditable before any graph write occurs. The PersonHarvestPlan is the audit trail.
Execution is resumable without re-reasoning. A timed-out harvest resumes from step N, not from the beginning.
New federation sources can be added by extending the context packet and the federation tier table in ADR-007. Layer 1 and Layer 3 require no changes for a new source whose data maps to existing relationship types.
Conflict resolution is structurally separate from profile construction. ConflictNote nodes accumulate without blocking the harvest of non-conflicting claims.
The agent model is replaceable without changing the graph schema. agent_model is recorded on the SYS_HarvestPlan for audit; a better model can re-reason the same context packet and produce a new plan.
