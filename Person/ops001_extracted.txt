1. Situation
The full DPRR dataset was imported 2026-02-25 via the live SPARQL endpoint at http://romanrepublic.ac.uk/rdf/endpoint/ using scripts/federation/dprr_import.py. All five phases completed successfully. The endpoint is now blocked by Anubis bot protection and is not accessible for programmatic queries.
This note records the accepted operational posture: treat the February 2026 import as a complete, stable baseline. Proceed with person schema enrichment against the static snapshot. No re-import attempt will be made until a data dump is obtained directly from King's Digital Lab (KDL).

3. StatusType Resolution  (closes ADR-007 OI-01)
Finding
The DPRR ontology rdfs:comment explicitly states the Status authority list "at present, only two are identified in this way: eques and senator." The Django codebase contains an add_senator_statusassertions management command that populates senator status ex-officio for persons holding qualifying posts (consul, praetor, aedile, tribunus plebis, censor, princeps senatus). This explains the asymmetric counts — senator is far more numerous than eques because it is derived from office-holding across the full corpus.

6. Wikidata Proxy for Drift Detection
While the DPRR endpoint is blocked, changes to the DPRR corpus can be partially detected via Wikidata through the P6863 (DPRR person ID) property. Run periodically against the Wikidata SPARQL endpoint.
SELECT ?person ?dprr_id ?label ?modified WHERE {
  ?person wdt:P6863 ?dprr_id .
  ?person rdfs:label ?label FILTER(lang(?label) = "en") .
  ?person schema:dateModified ?modified .
  FILTER(?modified > "2026-02-25T00:00:00Z"^^xsd:dateTime)
}
ORDER BY DESC(?modified)
What this detects
What it cannot detect

New DPRR persons without Wikidata QID alignment
Changes to existing Wikidata items (dates, offices, family links)
DPRR assertion corrections not yet reflected in Wikidata
New P6863 alignments for previously unlinked DPRR persons
Persons removed or merged in DPRR (deletions invisible via modification date)
