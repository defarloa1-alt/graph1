1. Session Context
This report covers decisions and artefacts produced in the design session following completion of the ADR-007 core specification. Starting point was the question of GEDCOM/standards alignment. Ending point is a calibrated Wikidata backlink capture design grounded in real P31 data from Pompey's backlink profile.


2. Standards Alignment  (ADR-007 Appendix)
GEDCOM and CIDOC-CRM alignment was added as Appendix A to ADR-007 rather than a separate ADR. Rationale: this is reference material that supplements the schema, not an architectural decision that changes it.
2.1  Standards evaluated

2.2  GEDCOM 7.0 key mapping decisions
The graph model is the source of truth. GEDCOM is a derived export — a projection of the graph, not the internal model. Three non-obvious mapping decisions:
No FAM node exists in the graph. Family units are derived at export time by finding shared FATHER_OF/MOTHER_OF targets. Ambiguous parentage produces multiple FAM candidates — flagged for review.
Tria nomina maps to GEDCOM NAME subtags imperfectly: praenomen → GIVN, nomen → SURN, cognomen → NICK. Gens and tribe use custom _GENS and _TRIBE extension tags.
BCE year offset: year N BCE = GEDCOM year (N+1) BC. 106 BCE → 0107 BC. Must be applied consistently at export — off-by-one errors corrupt all ancient dates.

2.3  CIDOC-CRM forward-looking event nodes
Full CRM compliance requires birth, death, marriage, and office holding as first-class event nodes. These are deferred per OI-02 but the target classes are documented in Appendix A.1.3 to guide future ADRs.

3. Person Harvest Agent Architecture  (ADR-008)
The central question: how much of the harvest pipeline should be agent-driven reasoning versus deterministic rule execution? The answer is a three-layer split. The agent operates only in the middle layer and never writes to the graph.
3.1  The core decision

The agent never writes directly to the graph. It receives a fixed context packet (pre-fetched graph state + federation raw data) and produces a PersonHarvestPlan. Layer 3 executes the plan deterministically. The plan is auditable and resumable — a timed-out harvest resumes from step N without re-reasoning.

3.2  Why agent is necessary for name reconciliation
The same person expressed across seven federation sources:

No deterministic string matcher reliably unifies these. The agent brings world knowledge to reason over the full name surface, date window, and cross-IDs. Once it resolves identity, the merge is deterministic.

3.3  Federation scope mismatch — a key agent task
A deterministic coverage matrix (source × attribute) would be brittle and require manual maintenance as new federations are added. The agent instead reasons about what each source should cover given its domain: VIAF silence on offices = scope mismatch; DPRR silence on a Wikidata P21 target = genuine absence. Different handlers, different confidence.

3.5  Family tree traversal strategy
Person-by-person construction, depth-first along paternal lines (DPRR filiation is primarily paternal), then maternal, then spouses, siblings, children. Five stopping conditions including: no DPRR record + no Wikidata P31=human confirmation → ScaffoldNode stub only; mythological persons (Romulus, Remus) are terminal nodes.
⚑  8-generation lineages already confirmed in graph (Cornelia (409), Calventia (B)). Average DPRR person has 2.2 family connections — deep trees are the exception.

5. Person Label Strategy  (ADR-007 Section 4 update)
The current label on DPRR persons (POMP1976 Cn. Pompeius (31) Cn. f. Sex. n. Clu. Magnus) is a technical key, not a display label. It conflates gens prefix, DPRR ID, tria nomina, and filiation into a single opaque string. Once these are parsed into first-class nodes per ADR-007, the label needs to be separated into four distinct properties.


label_sort is the property that makes family tree list views meaningful. Without it, list ordering defaults to English first name and scatters related persons. With it, all Pompeii sort together, cognomina within the gens are ordered, praenomen disambiguates last.
⚑  For DPRR-only persons awaiting Wikidata enrichment, label is temporarily set to label_latin. It is never null or empty.

6. Wikidata Backlink Capture Design
6.1  Design rationale
The family tree LLM harvest pass is expensive. Wikidata backlink capture is cheap and deterministic — it runs in Layer 1 alongside DPRR label parsing, before the LLM is invoked. Piggybacking it on the harvest pass avoids a second iteration over 2,856 QID-bearing persons later.
Two signals captured: (1) categorised backlink counts — a multi-dimensional prominence vector that drives harvest prioritisation and federation source selection; (2) raw person→person candidate list — seed for future relationship enrichment, stored as a JSON property for later agent classification.

6.2  What is not done during this pass
No WD_BacklinkCandidate nodes created
No relationship edges written from backlink data
No LLM classification of candidates
No blocking on Wikidata timeout — any failure returns None and harvest continues

6.3  Bucket taxonomy — calibrated from real P31 data
Initial classifier used six generic buckets. Running the P31 query against Pompey's actual backlinks (28 items, 43 distinct P31 values) revealed the generic buckets were too coarse. The dominant signal — numismatic — was entirely absent.


6.4  The numismatic finding
Coins and coin types account for 50 of Pompey's backlinks — more than humans, media, and scholarly reception combined. This was completely absent from the initial classifier. For Roman Republican prosopography this is a fundamental gap: numismatic evidence is a primary attestation type for the period.
Architectural implication: wd_bl_numismatic > 0 on a Person node triggers Nomisma as a priority source in the PersonHarvestPlan. The backlink profile drives federation source selection, not just enrichment depth.

6.6  SPARQL queries
Two queries defined for use from the harvest script or direct investigation:

Full backlink profile with per-item P31 (for harvest script Layer 1):
SELECT ?prop ?item ?itemLabel (GROUP_CONCAT(DISTINCT ?p31Label; separator=", ") AS ?types) WHERE {
  ?item ?prop wd:Q125414 .
  FILTER(STRSTARTS(STR(?prop), "http://www.wikidata.org/prop/direct/"))
  FILTER(?prop != schema:dateModified)
  FILTER(?prop != owl:sameAs)
  OPTIONAL { ?item wdt:P31 ?p31 . ?p31 rdfs:label ?p31Label FILTER(lang(?p31Label)="en") }
  SERVICE wikibase:label { bd:serviceParam wikibase:language "en" . }
}
GROUP BY ?prop ?item ?itemLabel
ORDER BY ?types ?itemLabel

Distinct P31 taxonomy with counts (for classifier calibration):
SELECT DISTINCT ?p31 ?p31Label (COUNT(DISTINCT ?item) AS ?cnt) WHERE {
  ?item ?prop wd:Q125414 .
  FILTER(STRSTARTS(STR(?prop), "http://www.wikidata.org/prop/direct/"))
  ?item wdt:P31 ?p31 .
  SERVICE wikibase:label { bd:serviceParam wikibase:language "en" . }
}
GROUP BY ?p31 ?p31Label
ORDER BY DESC(?cnt)

6.7  Integration point in harvest script
Drop into per-person loop after DPRR label parse, before LLM context packet assembly:
from wd_backlink_capture import capture_backlinks, build_update_cypher, RateLimiter
rl = RateLimiter()   # one instance for whole run — 3 req/s conservative

# Inside per-person loop:
if person["qid"] and person["qid"].startswith("Q"):
    rl.wait()
    bl = capture_backlinks(qid=person["qid"], label=person["label"])
    if bl:
        cypher, params = build_update_cypher(person["entity_id"], bl)
        session.run(cypher, params)
        # Optional: pass profile into LLM context packet
        context_packet["backlink_profile"] = {
            "total": bl["wd_bl_total"],
            "numismatic": bl["wd_bl_numismatic"],
            "persons": bl["wd_bl_persons"],
            "events": bl["wd_bl_events"],
        }

⚑  Persons where wd_bl_capped = true hit the 500-row LIMIT. Caesar, Cicero, Augustus will almost certainly cap. Flag for targeted follow-up query with property filtering after main pass.
